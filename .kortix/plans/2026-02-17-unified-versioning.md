**Updated:** 2026-02-17

**Goal:** One version number, one changelog, one release script across the entire Kortix Computer platform — sandbox, CLI, SDK, Docker image, GitHub releases, and the embedded Kortix CLI.

| Frontend | `0.1.0` | Vercel | auto-deploy |

**Problems:**
1. **Version sprawl** — 5 different version numbers for one platform
2. **No changelog** — updates show "Update to v0.4.17" with zero context
3. **No GitHub releases** — no tagged releases, no release notes
4. **Manual version bumping** — easy to forget one of the many places
5. **No enforcement** — can publish without changelog or tagging

### What We Built Previously

1. `sandbox/package.json` as single source of truth — Dockerfile + postinstall.sh read ALL versions from it
2. Live update system — `@kortix/sandbox` on npm → sandbox detects → user clicks update → postinstall.sh deploys everything
3. Zero hardcoded versions — Dockerfile and postinstall.sh both read from package.json
4. Full documentation — `docs/sandbox-update-system.md`

### Resolved Questions

1. **What is the "Kortix CLI"?** — It's the embedded bash CLI helper generated by `scripts/get-kortix.sh` at `~/.kortix/kortix`. It wraps `docker compose` commands (start, stop, update, etc.). Its VERSION is hardcoded at line 557 of `get-kortix.sh` — currently `0.2.0`. The release script auto-stamps it.

2. **Should CLI/SDK always re-publish?** — Yes. Simplest approach. `--skip-cli` and `--skip-sdk` flags for when they haven't changed.

3. **GH Workflow vs local script?** — **Local script.** Reason: all auth (npm, gh, docker) already exists locally. No GitHub secrets setup needed. Faster to debug. Only one person publishes. Can be wrapped in a GH Workflow later if needed.

4. **Docker image cadence?** — Optional via `--docker` flag. Most releases don't need Docker since live update handles everything deployable via postinstall.sh.

## All Artifacts & Release Surfaces

### 1. `@kortix/sandbox` (npm)
- The master update package. Contains agents, skills, tools, commands, plugin, kortix-master, configs, browser-viewer, postinstall.sh, CHANGELOG.json
- Published by: `npm publish` from `sandbox/`
- Version: Must match unified version

### 2. `@kortix/opencode-ai` (npm)
- The OpenCode CLI binary (11 platform variants, compiled via Bun)
- Published by: `KORTIX_VERSION=X.Y.Z bun ./script/publish-kortix.ts latest` from `services/opencode/packages/opencode/`
- Version: Must match unified version
- Note: Requires compilation. `--skip-cli` when source unchanged.

### 3. `@kortix/opencode-sdk` (npm)
- TypeScript SDK for the OpenCode REST API
- Published by: `KORTIX_SDK_VERSION=X.Y.Z bun ./script/publish-kortix.ts latest` from `services/opencode/packages/sdk/js/`
- Version: Must match unified version
- Note: `--skip-sdk` when API surface unchanged.

### 4. Docker Image (`kortixmarko/sandbox:X.Y.Z`)
- Full sandbox image — Alpine + webtop + all runtimes
- Published by: `sandbox/push.sh` → Daytona snapshot + Docker Hub
- Version: Read from `sandbox/package.json` at build time
- Note: `--docker` flag. Only needed when OS-level deps change.

### 5. GitHub Release (`vX.Y.Z`)
- Tagged release on `kortix-ai/computer` with auto-generated release notes from CHANGELOG.json
- Published by: `gh release create` (via release.sh)

### 6. Embedded CLI (`scripts/get-kortix.sh`)
- Bash CLI helper installed at `~/.kortix/kortix` via `curl get.kortix.ai/install`
- VERSION at line 557 — auto-stamped by release.sh via `sed`
- Commands: start, stop, restart, logs, status, update, version, etc.

### 7. Kortix API + Frontend
- Deployed independently (API via SSH on push to main, Frontend via Vercel)
- Not part of the unified version — they reference it but don't carry it

## Design

### The Version

One version: **`MAJOR.MINOR.PATCH`** (semver)

- **MAJOR:** Breaking changes (API, config format, data migration)
- **MINOR:** New features, significant updates
- **PATCH:** Bug fixes, dependency bumps

Current: `0.4.16` → next release: `0.5.0`

### The Changelog

**File:** `sandbox/CHANGELOG.json` (structured, machine-readable, bundled in npm package)

```json
[
  {
    "version": "0.5.0",
    "date": "2026-02-17",
    "title": "Unified versioning & upstream sync",
    "description": "Synced with OpenCode upstream v1.2.6, unified all dependency versioning.",
    "changes": [
      { "type": "feature", "text": "Live update now also updates CLI binary and agent-browser" },
      { "type": "feature", "text": "Changelog system — see what's new before updating" },
      { "type": "improvement", "text": "All versions read from sandbox/package.json" },
      { "type": "fix", "text": "Fixed s6 service path (s6-rc.d not services.d)" },
      { "type": "upstream", "text": "Synced with OpenCode v1.2.6" }
    ]
  }
]
```

**Change types:** `feature`, `fix`, `improvement`, `breaking`, `upstream`, `security`, `deprecation`

**Deployment chain:**
- Bundled in `@kortix/sandbox` npm package (via `files` array)
- `postinstall.sh` copies to `/opt/kortix/CHANGELOG.json`
- `Dockerfile` COPYs to `/opt/kortix/CHANGELOG.json`
- kortix-master reads it at `/kortix/health` and `/kortix/update` response
- Platform API fetches from GitHub raw URL (5min cache)
- Frontend shows in update UI

### The Release Script

**File:** `sandbox/release.sh` — runs locally with your existing auth

```
./sandbox/release.sh 0.5.0              # full release
./sandbox/release.sh --dry-run 0.5.0    # validate only
./sandbox/release.sh --skip-cli --skip-sdk 0.5.0   # sandbox-only
./sandbox/release.sh --docker 0.5.0     # include Docker
```

What it does:
1. **Checks prerequisites** — npm/gh/bun/node auth + installed
2. **Validates** — CHANGELOG.json has entry, version not already on npm
3. **Bumps versions** — `sandbox/package.json` (version + CLI dep), `get-kortix.sh` VERSION
4. **Builds + publishes CLI** → `@kortix/opencode-ai@0.5.0` (unless `--skip-cli`)
5. **Builds + publishes SDK** → `@kortix/opencode-sdk@0.5.0` (unless `--skip-sdk`)
6. **Publishes sandbox** → `@kortix/sandbox@0.5.0`
7. **Creates GitHub release** — `v0.5.0` with auto-generated notes from changelog
8. **Optionally builds Docker** — via existing `push.sh` (with `--docker`)
9. **Reminds to commit** — the version bump changes need committing

**Publish order matters:** CLI → SDK → Sandbox. Because `postinstall.sh` does `npm install -g @kortix/opencode-ai@{version}` — the CLI must exist on npm before any sandbox tries to update.

### Platform API

`GET /v1/platform/sandbox/version` — extended response:

```json
{
  "version": "0.5.0",
  "package": "@kortix/sandbox",
  "changelog": {
    "title": "Unified versioning & upstream sync",
    "description": "...",
    "date": "2026-02-17",
    "changes": [...]
  }
}
```

Changelog fetched from GitHub raw URL (not unpkg/CDN — under our control, already used by `get-kortix.sh`). 5-minute cache matches existing npm version cache.

New route: `GET /v1/platform/sandbox/version/changelog` — full history.

### Frontend

Update UI shows: version badge + "Update to vX.Y.Z" button + changelog title + first 3 changes.

## Task Breakdown

### Step 1: Create `sandbox/CHANGELOG.json`
- **NEW file** with backfilled version history (0.1.0 through 0.5.0)
- Newest first, each entry has version/date/title/description/changes[]

### Step 2: Wire changelog into package + deployment
- Add `"CHANGELOG.json"` to `sandbox/package.json` files array

- `sandbox/kortix-master/src/routes/update.ts` — add `changelog` field to update response

### Step 4: Extend Platform API
- `services/kortix-api/src/platform/routes/version.ts` — fetch changelog from GitHub raw URL, include in version response, add `/changelog` route

### Step 5: Frontend updates
- `apps/frontend/src/lib/platform-client.ts` — add ChangelogEntry types, extend SandboxVersionInfo/SandboxUpdateResult
- `apps/frontend/src/hooks/platform/use-sandbox-update.ts` — expose changelog in return value

## Risks & Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| Partial publish (CLI ok, sandbox fails) | Inconsistent npm state | Publish order: CLI → SDK → Sandbox. Script exits on failure. Re-run with `--skip-cli --skip-sdk`. |
| Version already on npm | Publish fails | Script checks `npm view` before publishing. Clear error message. |
| Changelog missing | Publish without context | Script refuses to run without valid changelog entry. |
| `get-kortix.sh` VERSION out of sync | Users see wrong version | Script auto-stamps via `sed`. |
| Developer forgets to commit version bump | Next dev sees stale version | Script reminds at end. Could add git check. |
| GitHub raw URL slow/down | Platform API can't serve changelog | 5-min cache + graceful fallback to empty changelog (version still returned). |
| npm publish triggers postinstall on dev machines | Dev laptop tries to deploy to `/opt/` | postinstall.sh already guards with `[ -d /opt ]` check. |
| SDK dep `^0.4.0` can't resolve `0.5.0` | Sandbox bun install fails | Step 6 bumps to `^0.5.0` before release. |
